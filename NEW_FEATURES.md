1. Auto-discover / auto-launch Excel with the debug port
The single biggest friction point. Users must manually launch Excel with WEBVIEW2_ADDITIONAL_BROWSER_ARGUMENTS="--remote-debugging-port=9222" every session, and scan_windows.go:11 is still a stub returning ErrNotFound. CLAUDE.md currently forbids auto-launch — but for an AI bridge, "Excel isn't reachable" is the most common failure mode and the AI can't recover from it. Implement Windows process scan + a --launch-excel flag that spawns Excel with the env var set. Even just a friendly diagnostic ("detected Excel running without --remote-debugging-port; relaunch with X") would help.

1. Drop the per-session exclusive lock; allow concurrent CDP calls
Session.Acquire serializes every tool call against the same session (internal/session/). CDP itself supports concurrent commands over flatten sessions natively — you only need a lock around connection establishment and reconnect, not steady-state command dispatch. For an AI that issues batched parallel calls (read range + list tables + screenshot), this is a real ceiling. Convert the session mutex to an RWMutex (write-lock only on dial/reconnect) or move to per-resource locking. Big win for "max performance" since the architecture already does the hard work (selector cache, target attach).

1. Make error envelopes actionable for the AI caller
The envelope has good structural diagnostics (cdpRoundTrips, durationMs, targetID) but error payloads are sparse. An AI that gets connection/session_acquire_failed: reconnect budget exhausted has nowhere to go. Enrich EnvelopeError.Details with: (a) the probed endpoint, (b) the last-known target list, (c) a recovery_hint field (e.g., "launch Excel with --remote-debugging-port=9222", "this addin manifest expects URL X, found Y"). The AI can self-recover from rich errors; it can't from terse ones. Low-effort, high-leverage.

1. Curate the 411 cdp.* tool surface
411 raw CDP tools cost the AI ~thousands of tokens per session in tool listings, and most are noise (Animation., WebAuthn., Debugger.*) for Office automation. Three changes: (a) keep --expose-raw-cdp off by default (already done), (b) introduce capability groups (--cdp-domains=DOM,Page,Runtime,Input) so users opt into the slice they need, (c) enrich descriptions on the high-level excel.* / page.* / addin.* tools with concrete examples in the schema — those are what the AI should reach for first. The codegen pipeline at cmd/gen-cdp-tools/ already has the manifest plumbing for this.

1. Structured logging + request correlation, with panic recovery
Today the server is silent unless --log-file is set, no slog, no request IDs, and the CDP readLoop goroutine has no panic recovery — a malformed frame can silently kill the connection. For a production AI bridge: (a) move to log/slog with JSON output, (b) thread a request ID through Dispatch → Session.Acquire → CDP commands and stamp it into diagnostics, (c) defer recover() in every long-lived goroutine with a re-dial, (d) optionally OpenTelemetry spans so you can see end-to-end latency breakdown (target discovery vs. CDP RTT vs. Office.js execution). This is what unlocks debugging at scale and gives you data to optimize against.
